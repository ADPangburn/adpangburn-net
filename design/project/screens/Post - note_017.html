The retry loop is a human-in-the-loop loop

Every retry budget should bottom out in a question to a person, not a RETRY_EXHAUSTED exception. Rewriting orchestrator/retry.ts on a Sunday because Monday's audit needed the trace.

orchestration hitl compliance

I spent the weekend rewriting how orchestrator/retry.ts handles tool failures. The short version: every retry budget should bottom out in a question to a person, not a RETRY_EXHAUSTED exception. Here is the trace from a real run — three loops, one human, one save.

This is the kind of refactor that looks small in the diff and feels enormous in production. We had three weeks of incident reports that all collapsed into the same shape, and I had stopped reading them because the title was always "agent failed silently." It turns out agents don't fail silently. They fail with all the information you need, addressed to nobody.

What was wrong

The old loop did the obvious thing: retry N times with exponential backoff, then raise. In a regulated environment that is not a feature, it is a future audit finding¹. The retries are cheap, the human is the expensive resource, and the human is the only thing in the system that knows whether this particular call should be retried, redirected, or abandoned.

The failure mode I kept hitting was: the agent burns three retries on a flaky internal tool, surfaces a stack trace to a customer-facing service, the on-call gets paged, and our SOC2 evidence is a screenshot. The agent did not ask. It just gave up.

caveat
This only works because we shipped the trace viewer first. If you cannot see what the agent tried, you cannot ask a useful question about it. See note_015 for why the trace viewer is post zero.

The new shape

The change is fifteen lines of code. Most of it is removing the exception class.

orchestrator/retry.tstypescript
// retries bottom out in a human, not an exception
export async function retry(task: Task, budget = 3) {
  for (let attempt = 0; attempt < budget; attempt++) {
    try { return await task.run() }
    catch (e: ToolError) {
      await logAttempt(task.id, e, attempt)
    }
  }
  // budget exhausted → bottom out in a human, not an exception
  return askHuman(task, {
    reason: "out of retries",
    attempts: await loadAttempts(task.id),
    severity: task.tier === "prod" ? "page" : "queue",
  })
}
audit note
Every askHuman call writes a row into the decision log with the attempt history, the human who answered, and what they chose. Compliance reads this table on Tuesdays. The table schema is in the appendix of our SOC2 doc.

What falls out of this

Three things, in order of how much they matter:

  1. The agent now has a vocabulary for "I am stuck." That is the API the human consumes. Before, the agent's only escape hatch was raising. Now it has a typed message.
  2. Logs from each attempt are passed through to the human. They do not have to re-derive what was tried. This collapsed our MTTR on agent-related incidents from forty minutes to under five.
  3. The next time the agent hits this class of failure, it tries the human's answer first. We keep a thin lookup table keyed on (task.type, error.code). It is not memory, it is procedural taste.

The lookup table is not memory

I want to be careful here because "agent that learns from human feedback" is the kind of phrase that gets a feature shipped to production and a compliance officer fired. The lookup table is not learning. It is a cache. Every entry has an explicit author, an expiry, and a one-click "this is stale" button. Compliance signed off on it because it is, structurally, a runbook — just one that the agent reads instead of the on-call.

What I am not claiming

I am not claiming this scales to every kind of failure. Some failures should still raise: anything that violates an invariant, anything that touches money, anything that the human can't reasonably triage in under a minute. The point is not "no exceptions ever." The point is that "out of retries" is not an invariant violation — it's a request for help, and the system should treat it like one.

tl;dr
Retries are cheap. Humans are the expensive resource. Audit logs are the only feature compliance cares about. If your agent loop can raise, you have designed a system that fails closed when it should fail to a human. Bottom out the loop.
The retry budget is a confidence interval on the tool. The human is the prior. Stop modeling the system as if the tool were the prior.
